# Алгоритмы управления

## 1. Маршрутизация товара

```
Вход: штрихкод
1. Сканирование → barcode
2. WMS.resolve(barcode) → destination (1..400)
3. routing.select_destination(destination):
   a. если ячейка доступна → destination
   b. иначе поиск в том же модуле с min(queue_depth)
   c. иначе глобальный overflow с min(queue_depth)
   d. иначе reject (потеря)
4. Индукция на модуль ceil(destination / 100)
5. Транспорт → сброс → КТЯ
```

## 2. Антизатор (anti-jam)

Пороги:
- `chute_buffer_capacity = 12` — макс. очередь на ячейку
- `jam_threshold_queue = 8` — мягкий порог перенаправления
- `kty_fill_ratio >= 0.98` — ячейка недоступна

При перегрузке товар направляется в ближайшую свободную ячейку **того же модуля**, затем — глобальный overflow.

## 3. Управление КТЯ

Цикл для каждой ячейки (процесс `kty_manager`):

```
каждую 1 с:
  fill_ratio = fill / capacity
  если fill_ratio >= 0.88 и random < 0.15:
    уплотнение (×0.92 объёма), 0.4 с
  если fill >= capacity:
    блокировка ячейки
    смена КТЯ (14 с автомат / 45 с ручная)
    fill = 0
```

## 4. Балансировка входного потока

`InputBalancer` отслеживает загрузку модулей:
- `active_items` — товары в обработке на модуле
- `induction_queue` — ожидающие индукцию

На входном распределителе (upstream) выбирается модуль с минимальным `score`.  
После назначения WMS модуль **фиксирован** — товар нельзя отправить на другой модуль.

## 5. Нештатные режимы

| Событие | Алгоритм |
|---------|----------|
| Отказ ячейки | `state = FAILED`, overflow в модуле |
| Деградация модуля | `transit_time × 1.35`, флаг в balancer |
| Перегрузка 125% | очереди растут, растёт misroute_rate |
| Hotspot | 55% потока в модуль 1 → рост перенаправлений |

## 6. WMS-интерфейс

HTTP-имитатор (`control/wms_server.py`):

```
GET /health
GET /route/{barcode}
→ {"barcode", "destination", "route_hint"}
```

Реализация: `control/wms_mock.py` — детерминированный hash(barcode) % 400 + 1.
